---
title: "Secret label reference for vector-config operator"
description: "Complete reference for the three labels that control which Kubernetes Secrets vector-config selects, syncs as config files, and syncs as TLS certificates."
---

vector-config uses Kubernetes Secret labels as the sole mechanism for selecting which Secrets to process. You apply labels directly to your Secret manifests — no other configuration is required to enroll a Secret. This page documents every label key and value that the operator recognizes, how they combine, and the file paths they produce.

<Warning>
  All label keys and values are case-sensitive. The role value must be exactly `Agent` or `Aggregator` (capital first letter). The boolean-style values `"true"` must be the string `true`, not `True`, `TRUE`, or `1`.
</Warning>

## Label reference

<ParamField path="logging.whizard.io/vector-role" type="string" required>
  Identifies the operator role that should process this Secret. Must be exactly `Agent` or `Aggregator`. The operator compares this value against its own `--role` flag at reconcile time — a mismatch means the Secret is ignored.
</ParamField>

<ParamField path="logging.whizard.io/enable" type="string">
  When set to `"true"`, includes this Secret in the configuration file sync. The operator writes each data key in the Secret as a file named `<secret-name>-<data-key>` under `FileDir`. Omitting this label, or setting any value other than `"true"`, excludes the Secret from config sync.
</ParamField>

<ParamField path="logging.whizard.io/certification" type="string">
  When set to `"true"`, includes this Secret in the TLS certificate sync. The operator writes each data key in the Secret as a file named `<data-key>` under `<FileDir>/certification/`. Omitting this label, or setting any value other than `"true"`, excludes the Secret from cert sync.
</ParamField>

## How the labels combine

A Secret must carry `logging.whizard.io/vector-role` plus exactly one of the secondary labels to be processed. The two secondary labels are mutually exclusive in practice: a Secret should be either a config Secret or a cert Secret, not both.

| Purpose | Required labels |
|---|---|
| Configuration file sync | `vector-role=<role>` + `enable="true"` |
| TLS certificate sync | `vector-role=<role>` + `certification="true"` |

## Configuration Secret example

This Secret is picked up by an Agent operator instance and written as a config file.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: agent-pipeline
  namespace: kubesphere-logging-system
  labels:
    logging.whizard.io/vector-role: Agent      # target the Agent operator instance
    logging.whizard.io/enable: "true"           # include in config file sync
type: Opaque
data:
  vector.yaml: <base64-encoded Vector config YAML>
```

The data key `vector.yaml` in the Secret named `agent-pipeline` produces this file on disk:

```
/etc/vector/custom/agent-pipeline-vector.yaml
```

If the Secret has multiple data keys, each key produces a separate file:

```
/etc/vector/custom/agent-pipeline-sources.yaml
/etc/vector/custom/agent-pipeline-sinks.yaml
```

## TLS certificate Secret example

This Secret is picked up by an Aggregator operator instance and written into the `certification/` subdirectory.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: vector-ca
  namespace: kubesphere-logging-system
  labels:
    logging.whizard.io/vector-role: Aggregator  # target the Aggregator operator instance
    logging.whizard.io/certification: "true"     # include in TLS certificate sync
type: Opaque
data:
  ca.crt: <base64-encoded CA certificate>
  tls.crt: <base64-encoded TLS certificate>
  tls.key: <base64-encoded TLS private key>
```

Each data key is written directly under `certification/` — the Secret name is **not** included in the filename:

```
/etc/vector/custom/certification/ca.crt
/etc/vector/custom/certification/tls.crt
/etc/vector/custom/certification/tls.key
```

<Note>
  Because cert file names come only from the data key, two CA Secrets with the same data key will overwrite each other's output file. Use unique data key names across all CA Secrets for the same role.
</Note>

## File naming rules

| Secret type | Output path pattern |
|---|---|
| Config (`enable: "true"`) | `<FileDir>/<secret-name>-<data-key>` |
| Certificate (`certification: "true"`) | `<FileDir>/certification/<data-key>` |

The default `FileDir` is `/etc/vector/custom`. You can override it with the `--fileDir` flag. See [Architecture](/concepts/architecture) for the full reconcile flow.

## Aggregator config Secret example

This Secret is processed by an Aggregator operator instance and written as a config file.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: aggregator-pipeline
  namespace: kubesphere-logging-system
  labels:
    logging.whizard.io/vector-role: Aggregator  # target the Aggregator operator instance
    logging.whizard.io/enable: "true"            # include in config file sync
type: Opaque
data:
  aggregator.yaml: <base64-encoded Vector config YAML>
```

This produces:

```
/etc/vector/custom/aggregator-pipeline-aggregator.yaml
```
