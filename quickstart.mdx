---
title: "Deploy vector-config and ship your first Vector config"
description: "Install the vector-config operator, label a Kubernetes Secret, and verify that the operator writes Vector configuration files to disk in under ten minutes."
---

vector-config is a Kubernetes operator that watches labeled Secrets and writes their data as configuration files to `/etc/vector/custom` (or a path you choose). This page walks you through building and pushing the operator image, deploying it to your cluster, creating a labeled Secret, and confirming that the operator wrote the expected files — everything you need to have a working installation.

<Note>
  vector-config does not install or run Vector itself. You must run Vector separately as a DaemonSet or Deployment and point it at the same directory the operator writes to.
</Note>

## Prerequisites

Before you start, make sure you have:

- A Kubernetes cluster with `kubectl` configured
- Docker (or a compatible container tool) with push access to your image registry
- `kustomize` v5.0.1 or later
- Go 1.20 or later (only needed if you build from source)

## Installation

<Steps>
  <Step title="Clone the repository">
    Clone the vector-config source and enter the project directory.

    ```bash
    git clone https://github.com/WizTelemetry/vector-config.git
    cd vector-config
    ```
  </Step>

  <Step title="Build and push the operator image">
    The `docker-build` target runs tests, then builds and pushes a multi-arch image (`linux/amd64` and `linux/arm64`) using Docker Buildx. Set the `IMG` variable to your registry path.

    ```bash
    make docker-build IMG=your-registry/vector-config:latest
    ```

    <Tip>
      The Makefile defaults to `IMG=kubesphere/vector-config:latest`. Override it on the command line or by exporting the variable before running `make`.
    </Tip>

    If you only want a single-platform local build without running tests, you can call Docker directly:

    ```bash
    docker build -f Dockerfile -t your-registry/vector-config:latest .
    docker push your-registry/vector-config:latest
    ```
  </Step>

  <Step title="Deploy the operator">
    The `deploy` target patches the manager image in `config/manager/` and applies the full `config/default/` Kustomize stack.

    ```bash
    make deploy IMG=your-registry/vector-config:latest
    ```

    The operator deploys to the `vector-config-system` namespace and by default watches Secrets in `kubesphere-logging-system`, running in `Agent` mode and writing files to `/etc/vector/custom`. See [configuring the operator](#configuring-the-operator) to change these defaults.

    Confirm the operator pod is running:

    ```bash
    kubectl get pods -n vector-config-system
    ```

    ```
    NAME                                               READY   STATUS    RESTARTS   AGE
    vector-config-controller-manager-xxxxxxxx-xxxxx   1/1     Running   0          30s
    ```
  </Step>

  <Step title="Create a labeled Secret">
    The operator reconciles Secrets that carry two labels:

    | Label | Required value |
    |---|---|
    | `logging.whizard.io/vector-role` | `Agent` or `Aggregator` (must match the operator's `--role` flag) |
    | `logging.whizard.io/enable` | `"true"` |

    Apply the sample Agent Secret from the repository:

    ```bash
    kubectl apply -f config/sample/secret2.yaml
    ```

    Or create your own Secret with a Vector configuration payload. The Secret data keys become part of the written filenames (the operator writes each key as `<secret-name>-<key>`):

    ```yaml
    apiVersion: v1
    kind: Secret
    metadata:
      name: agent
      namespace: kubesphere-logging-system
      labels:
        logging.whizard.io/vector-role: Agent
        logging.whizard.io/enable: "true"
    data:
      agent.yaml: <base64-encoded Vector config YAML>
    ```

    <Note>
      The `logging.whizard.io/vector-role` label value must exactly match the role the operator was started with (`Agent` or `Aggregator`). Secrets with a mismatched or missing role label are ignored.
    </Note>
  </Step>

  <Step title="Verify the operator wrote the config files">
    After the Secret is applied, the operator reconciles immediately and writes the decoded data to disk. Each key in `secret.data` is written as a file named `<secret-name>-<key>` inside the config directory.

    For the `agent` Secret with key `agent.yaml`, the operator writes:

    ```
    /etc/vector/custom/agent-agent.yaml
    ```

    Use `kubectl exec` to inspect the files inside the operator pod:

    ```bash
    kubectl exec -n vector-config-system \
      deploy/vector-config-controller-manager -- \
      ls /etc/vector/custom/
    ```

    You should see the generated config file listed:

    ```
    agent-agent.yaml
    ```

    To read its contents:

    ```bash
    kubectl exec -n vector-config-system \
      deploy/vector-config-controller-manager -- \
      cat /etc/vector/custom/agent-agent.yaml
    ```

    <Tip>
      If the file does not appear, check the operator logs for reconciliation errors:

      ```bash
      kubectl logs -n vector-config-system deploy/vector-config-controller-manager
      ```
    </Tip>
  </Step>
</Steps>

## Configuring the operator

The operator binary accepts the following flags. You can set these in your Kustomize patch or Deployment spec.

| Flag | Default | Description |
|---|---|---|
| `--role` | `Agent` | Role to run as. Accepts `Agent` or `Aggregator`. Determines which Secrets the operator reconciles. |
| `--namespace` | `kubesphere-logging-system` | Namespace the operator watches for labeled Secrets. |
| `--fileDir` | _(empty, falls back to `/etc/vector/custom`)_ | Directory where config files are written. |
| `--metrics-bind-address` | `:8082` | Address the metrics endpoint listens on. |
| `--health-probe-bind-address` | `:8081` | Address the health probe endpoint listens on. |
| `--leader-elect` | `false` | Enable leader election for HA deployments. |

<CodeGroup>

```bash Agent mode (default)
/manager \
  --role=Agent \
  --namespace=kubesphere-logging-system \
  --fileDir=/etc/vector/custom
```

```bash Aggregator mode
/manager \
  --role=Aggregator \
  --namespace=kubesphere-logging-system \
  --fileDir=/etc/vector/custom
```

</CodeGroup>

<Warning>
  If you pass an invalid value for `--role`, the operator silently falls back to `Aggregator` mode. Double-check your Deployment spec if Secrets are not being reconciled.
</Warning>

## How the operator reconciles Secrets

On every reconcile loop, the operator:

1. Removes all existing files under the config directory (a full replace, not a merge).
2. Lists all Secrets in the watched namespace that match both `logging.whizard.io/vector-role: <role>` and `logging.whizard.io/enable: "true"`.
3. Writes each Secret data entry as a file at `<fileDir>/<secret-name>-<key>`.
4. Lists Secrets with the `logging.whizard.io/certification: "true"` label and writes their data into `<fileDir>/certification/<key>`.

<Note>
  Because the operator removes all files before writing, deleting a Secret or removing the required labels causes its config files to be cleaned up on the next reconcile.
</Note>

## Uninstalling

To remove the operator and its RBAC resources from the cluster:

```bash
make undeploy
make uninstall
```

Pass `ignore-not-found=true` if resources were already partially removed:

```bash
make undeploy ignore-not-found=true
make uninstall ignore-not-found=true
```
